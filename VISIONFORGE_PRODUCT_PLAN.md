## 1. 한눈에 보는 제품 기획

### 1.1 제품 정의

**[확정]** VisionForge는 사용자가 탐지할 대상의 이미지 여러 장을 등록하면, 로컬에서 학습 이미지를 합성·증강하고 객체 탐지 모델을 학습한 뒤, 새로운 이미지 여러 장에서 대상의 존재 여부, 개수, 위치, 신뢰도를 일괄 판정하는 Windows 및 macOS 데스크톱 프로그램이다.

VisionForge에서 말하는 분류는 별도 이미지 분류 모델로 이미지 전체에 클래스를 붙이는 기능이 아니다. 객체 탐지 결과를 바탕으로 대상 있음·없음, 여러 개 있음, 낮은 신뢰도, 검토 필요 등의 이미지 상태를 판정하는 기능을 의미한다.

핵심 가치는 다음 네 가지다.

1. 인터넷 없이 설치 이후의 모든 핵심 기능이 동작한다.
2. 단일 이미지를 포함한 모든 입력을 이미지 목록으로 처리한다.
3. 생성부터 추론까지 이미지별 상태, 오류, 이력을 독립적으로 관리한다.
4. 학습 모델과 추론 규칙을 하나의 패키지로 내보내 다른 컴퓨터에서 재학습 없이 사용한다.

### 1.2 권고 MVP 기준선

**[제안]** MVP는 아래 흐름에 집중한다.

1. 프로젝트당 대상 클래스 1개를 만들고 원본 대상 이미지 여러 장을 등록한다.
2. 구조화된 상황 옵션으로 합성 조건을 지정한다.
3. 로컬 2D 합성 및 일반 이미지 증강으로 양성·부정 학습 이미지를 여러 장 생성한다.
4. 합성 과정의 기하 정보를 이용해 Bounding Box를 자동 생성하고 사용자가 검토·수정한다.
5. 승인된 이미지로 변경 불가능한 데이터셋 버전을 만든다.
6. 로컬 객체 탐지 모델을 학습하고 고정 평가 세트로 성능을 비교한다.
7. 이미지 여러 장을 일괄 추론하고 오탐·미탐·저신뢰 결과를 검토한다.
8. 검토 결과를 새 데이터셋 버전에 반영해 반복 학습한다.
9. 모델, 클래스, 전처리, 후처리, 호환성, 무결성 정보를 .vfmodel 패키지로 내보낸다.
10. 다른 Windows 또는 macOS 컴퓨터에서 패키지를 가져와 즉시 추론한다.

### 1.3 가장 중요한 설계 결정

| 주제 | 기획 결정 |
|---|---|
| 처리 단위 | ImageSet과 Job을 1급 개념으로 두고 단일 이미지도 항목 1개의 ImageSet으로 취급 |
| 사용자 통제 | 품질 경고, 제외 추천, 자동 검사는 제공하되 원본을 임의 삭제하지 않음 |
| 오프라인 | 런타임, 기본 가중치, 디코더, 학습·추론 엔진을 설치 패키지 또는 오프라인 확장 패키지에 포함 |
| 생성 방식 | MVP는 재현 가능한 2D 합성·증강 우선, 로컬 확산 모델은 제외 |
| 자연어 | MVP는 구조화된 옵션 우선, 로컬 LLM은 제외 |
| 데이터 이력 | 원본부터 결과, 데이터셋, 모델까지 파생 관계와 버전을 추적 |
| 실행 구조 | UI와 장시간 AI 작업을 분리하고 작업 상태를 영속화 |
| 모델 이식성 | 범용 모델 형식과 VisionForge 전용 메타데이터 패키지를 결합 |
| 추론 기준 | CPU 호환 모드를 기준선으로 두고 가속 모드는 기능적 동일성을 검증 |
| 배포 | 운영체제·아키텍처별 완전 오프라인 설치 패키지와 파일 기반 업데이트 제공 |

## 2. 요구사항 분류

### 2.1 확정 요구사항

| 영역 | 확정 내용 | MVP 수용 기준 |
|---|---|---|
| 플랫폼 | Windows, macOS 지원 | 두 운영체제의 정식 설치 패키지로 핵심 흐름 완료 |
| 데이터 형식 | AI가 직접 처리하는 데이터는 이미지로 제한 | 비이미지 입력 경로와 실시간 스트림 기능이 없음 |
| 다중 이미지 | 등록, 생성, 검토, 데이터셋, 추론, 결과, 내보내기 모두 다중 이미지 기본 | 이미지 1장과 N장이 같은 작업 API와 상태 모델 사용 |
| 객체 탐지 | 대상 존재 여부, 개수, 위치, Bounding Box, 신뢰도 제공 | 이미지별 탐지 결과와 오류가 독립 저장됨 |
| 오프라인 | 외부 API, 원격 학습, 원격 큐, 필수 인증 서버 금지 | 네트워크 차단 상태에서 전체 MVP 시나리오 완료 |
| 로컬 런타임 | 사용자가 Python이나 AI 도구를 별도 설치하지 않음 | 설치 후 GUI만으로 실행 |
| 학습 | 로컬 CPU 학습 및 장치별 가속 가능성 검토 | CPU 기준 경로가 항상 존재하고 중단·실패 상태가 보존됨 |
| 추론 | 여러 이미지를 일괄 추론 | 일부 이미지 실패 시 나머지 항목 계속 처리 |
| 모델 공유 | Windows와 macOS 간 모델 공유, 재학습 없는 즉시 추론 | 내보내기·가져오기·무결성·호환성 검사를 통과 |
| 재현성 | 전처리와 후처리까지 패키지에 포함 | 동일 입력에 기능적으로 같은 탐지 결과 제공 |
| 저장 | 이미지, 데이터셋, 모델, 결과, 로그를 로컬에 저장 | 외부 자동 업로드나 동의 없는 원격 로그가 없음 |
| 안정성 | 자동 저장, 진행 상태 복구, 프로젝트 손상 방지 | 비정상 종료 후 완료 항목을 재실행하지 않고 재개 가능 |

### 2.2 기획 가정

| 가정 | 영향 | 확인 시점 |
|---|---|---|
| **[가정]** MVP는 로컬 단일 사용자 환경이다. | 계정, 조직 권한, 동기화 기능을 제외할 수 있음 | 제품 정책 확정 |
| **[가정]** 프로젝트당 클래스는 1개지만 내부 데이터 모델은 다중 클래스를 허용한다. | 후속 버전에서 데이터 이전 없이 확장 가능 | 데이터 모델 설계 검토 |
| **[가정]** 사용자는 등록 이미지와 배경 이미지의 사용 권한을 보유한다. | 앱은 권리 고지와 출처 기록만 제공 | 법무 검토 |
| **[가정]** 설치 용량보다 오프라인 완전성과 호환성을 우선한다. | CPU 런타임과 기본 모델을 기본 설치에 포함 가능 | 배포 PoC |
| **[가정]** CPU 학습은 느려도 기능적으로 제공한다. | 시간 예측보다 단계, 처리량, Epoch, 체크포인트를 강조 | 학습 벤치마크 |
| **[가정]** 원본 파일은 프로젝트 내부로 복사해 보존한다. | 외부 경로 이동·삭제에도 프로젝트가 재현 가능 | 저장 정책 확정 |

### 2.3 미확정 의사결정

| 항목 | 필요한 결정 |

| 입력 형식 | JPEG, PNG 외 WebP, BMP, TIFF, HEIC, AVIF의 교차 플랫폼 디코딩 범위 |
| 용량 제한 | 이미지 개수, 최대 픽셀 수, 프로젝트 크기의 경고·차단 기준 |
| macOS 범위 | Apple Silicon 필수, Intel Mac 지원 여부와 최소 macOS 버전 |
| 가속 패키지 | NVIDIA 및 Apple Silicon 가속 런타임을 기본 포함할지 확장 패키지로 분리할지 |
| 보안 | 모델·프로젝트 암호화, 비밀번호, 조직 인증서, 디지털 서명의 MVP 범위 |
| 패키지 확장자 | .vfmodel 및 프로젝트·업데이트 패키지 확장자 최종 확정 |
| 성능 기준 | 대상 유형별 정확도, 최소 이미지 수, 허용 추론 시간의 제품 기준 |

## 3. 범위

### 3.1 MVP 포함

| 제품 영역 | 포함 기능 |
|---|---|
| 프로젝트 | 생성, 열기, 자동 저장, 백업, 복원, 단일 대상 클래스 |
| 이미지 입력 | 다중 파일, 폴더, 드래그 앤 드롭, 추가 등록, 저장 세트 재사용 |
| 이미지 검사 | 손상, 형식, 크기, 해상도, 중복·유사, 블러, 대상 크기 관련 경고 |
| 전경 준비 | 로컬 배경 제거 또는 투명 영역 생성, 사용자 확인·수정 |
| 조건 설정 | 배경, 조명, 위치, 크기, 회전, 기울기, 가림, 개수, 노이즈, 블러 등 구조화 옵션 |
| 데이터 생성 | 2D 합성, 증강, 부정 이미지, 다중 생성, 랜덤 시드, 자동 Bounding Box |
| 검토 | 갤러리·목록, 다중 선택, 승인·제외, 재생성, 라벨 수정 |
| 데이터셋 | 학습·검증·테스트 분할, 불변 버전, 출처·라벨·시드 추적 |
| 학습 | 로컬 학습, 기본·고급 설정, 진행률, 중단, 실패 정보, 체크포인트 |
| 평가 | 탐지 지표, 실제 이미지 평가 세트, 이전 모델 비교 |
| 추론 | 다중 입력, 이미지별 결과·오류, 목록·갤러리, 재추론, 모델 비교 |
| 피드백 | 오탐·미탐·저신뢰 지정, 재학습 후보, 새 데이터셋 버전 |
| 내보내기 | 결과 이미지, CSV, JSON, 선택 결과 묶음 |
| 모델 공유 | .vfmodel 내보내기·가져오기, 체크섬·호환성 검사, CPU 폴백 |
| 오프라인 운영 | 전체 설치, 로컬 업데이트, 로컬 로그, 작업 복구 |

### 3.2 MVP 제외

**[확정]** 다음 항목은 초기 MVP에서 제외할 수 있다.

- 다중 클래스 동시 학습
- 동영상, 웹캠, 실시간 카메라·화면·스트림
- 음성, 문서, PDF, 센서 데이터 처리
- Segmentation, Rotated Bounding Box, Keypoint를 제품 출력 라벨로 제공하는 기능
- 3D 합성 및 로컬 생성형 이미지 모델
- 로컬 LLM 및 외부 LLM API
- 클라우드 학습·추론·저장·협업
- 사용자 계정, 권한, 동기화, 공동 편집
- 완전 자동 Active Learning
- 별도 추론 전용 앱
- 비트 단위로 완전히 동일한 교차 플랫폼 추론

내부 배경 제거를 위한 마스크는 합성 도구의 중간 데이터이며, 제품이 제공하는 학습 라벨의 Segmentation 기능과 구분한다.

## 4. 사용자와 전체 흐름

### 4.1 주요 사용자

| 사용자 | 주요 목표 |
|---|---|
| 학습·관리 사용자 | 대상 등록, 합성 조건 설정, 데이터 검토, 모델 학습·평가·공유 |
| 추론 사용자 | 공유 모델 가져오기, 이미지 일괄 판정, 결과 검토·내보내기 |

MVP는 하나의 앱에서 두 역할을 모두 제공하되, 시작 화면과 권한 체계를 별도 제품처럼 나누지 않는다.

### 5.3 프로세스 경계

**[제안]** UI 프로세스, 작업 관리자, 장시간 AI 작업 프로세스를 분리한다.

- UI가 종료되어도 마지막 커밋 지점까지 작업 상태가 보존된다.
- 학습 또는 추론 프로세스가 충돌해도 프로젝트 메타데이터와 UI가 함께 종료되지 않는다.
- 작업 프로세스는 요청된 프로젝트 디렉터리와 임시 디렉터리에만 접근한다.
- 작업 이벤트는 시작, 항목 진행, 항목 완료, 항목 실패, 체크포인트, 취소 확인, 작업 완료로 표준화한다.
- 작업 프로세스가 비정상 종료되면 실행 중 항목만 재검증하고 완료 항목은 재사용한다.
- GPU 메모리를 점유하는 작업은 기본적으로 한 번에 하나만 실행하고 CPU 작업은 자원 한도 안에서 병렬화한다.

## 6. 다중 이미지 작업 모델

### 6.1 통합 원칙

**[확정]** 모든 이미지 기능은 ImageSet을 입력받고 ImageSet 또는 이미지별 결과 목록을 반환한다.

| 입력 상황 | 내부 표현 |
|---|---|
| 이미지 1장 선택 | 항목 1개를 가진 ImageSet |
| 이미지 N장 선택 | 항목 N개를 가진 ImageSet |
| 폴더 선택 | 탐색과 형식 검사를 거쳐 만든 ImageSet |
| 이전 결과 재사용 | 기존 자산 ID를 참조하는 새 ImageSet |

별도 단일 이미지 처리 경로를 만들지 않는다. 미리보기와 즉시 실행도 동일한 작업 구조를 사용하되 우선순위만 높인다.

### 6.2 작업과 항목 상태

~~~mermaid
stateDiagram-v2
    [*] --> Queued
    Queued --> Running
    Running --> Succeeded
    Running --> Failed
    Running --> Pausing
    Pausing --> Paused
    Paused --> Queued
    Failed --> Queued: 재시도
    Queued --> Cancelled
    Running --> Cancelled: 안전 지점 확인 후
    Succeeded --> [*]
    Cancelled --> [*]
~~~

Job은 전체 작업 상태를, JobItem은 이미지별 상태를 저장한다. Job의 진행률은 JobItem 집계로 계산하며 임의의 단일 카운터를 진실의 원천으로 사용하지 않는다.

### 6.3 복구와 멱등성 규칙

1. 각 항목은 입력 자산 ID, 파라미터 해시, 엔진 버전, 랜덤 시드를 가진다.
2. 같은 입력과 설정으로 완료된 출력이 무결하면 재개 시 재사용한다.
3. 출력 파일은 임시 이름으로 완성한 뒤 원자적으로 최종 이름으로 교체한다.
4. DB에는 파일 완성 이후 완료 상태를 기록한다.
5. 취소는 안전 지점에서 확인하고 이미 완료된 항목은 유지한다.
6. 실패 항목은 오류 코드, 사용자 메시지, 상세 로그, 재시도 가능 여부를 각각 저장한다.
7. 앱 시작 시 실행 중이던 작업을 중단됨 상태로 전환하고 체크포인트에서 복구 가능 여부를 판단한다.
8. 디스크 공간 부족과 메모리 부족은 작업 시작 전과 실행 중 모두 감시한다.

### 7.3 이미지 역할

ImageAsset 하나의 구조로 아래 역할을 표현한다.

- target_original
- target_foreground
- background
- generated_positive
- generated_negative
- real_training
- validation
- test
- inference_input
- rendered_result

역할은 파일 복사를 의미하지 않는다. 같은 물리 파일을 여러 데이터셋이 참조할 수 있고, 데이터셋 버전은 참조와 라벨의 스냅샷을 고정한다.

### 7.4 불변성과 추적성

- 원본 자산은 가져온 이후 덮어쓰지 않는다.
- 편집 결과는 새 파생 자산으로 만든다.
- 데이터셋 버전은 생성 후 변경하지 않는다.
- 모델 버전은 정확히 하나의 데이터셋 버전과 학습 설정 스냅샷을 참조한다.
- 추론 결과는 사용 모델, 전처리·후처리 버전, 실행 장치를 기록한다.
- 사용자 수정은 원본 탐지값을 지우지 않고 수정 전·후 값을 함께 남긴다.
- 제외와 삭제는 구분한다. 제외는 논리 상태이며, 영구 삭제는 명시적 확인과 참조 검사 후 수행한다.

## 9. 이미지 등록과 품질 검사

### 9.1 등록 파이프라인

1. 파일·폴더·드래그 앤 드롭 입력을 하나의 수집 목록으로 만든다.
2. 확장자가 아닌 실제 파일 헤더와 디코딩 성공 여부를 검사한다.
3. EXIF 방향을 정규화하고 색상 프로파일·알파 처리 정책을 적용한다.
4. 파일 체크섬으로 완전 중복을, 지각 해시로 유사 이미지를 그룹화한다.
5. 크기, 픽셀 수, 해상도, 블러, 노출, 대상 크기 관련 품질 신호를 계산한다.
6. 결과를 오류, 강한 경고, 권고, 정상으로 표시한다.
7. 사용자가 이미지별 또는 그룹별 포함·제외를 결정한다.
8. 승인된 파일을 내부 저장소로 복사하고 원본 출처를 기록한다.

품질 점수는 학습 적합성의 절대 판정이 아니라 검토 우선순위다. 자동 검사 결과 때문에 파일을 영구 삭제하지 않는다.

### 9.2 형식 제안

| 형식 | 제안 상태 | 이유 |
|---|---|---|
| JPEG, PNG | **[제안] MVP 필수** | 일반성, 교차 플랫폼 디코더 안정성 |
| WebP, BMP | **[제안] MVP 후보** | 일반 이미지 수용 범위 확대, 디코더 검증 필요 |
| TIFF | **[미확정]** | 다중 페이지·고비트·색상 프로파일 정책 필요 |
| HEIC, AVIF | **[미확정]** | OS·코덱·재배포 라이선스와 동일 디코딩 검증 필요 |
| GIF | **[제안] 정지 첫 프레임만 또는 제외** | 애니메이션은 현재 범위와 충돌 |

최대 이미지 개수와 해상도는 임의의 고정값으로 먼저 제한하지 않는다. PoC에서 메모리, 디스크, 썸네일 성능을 측정한 뒤 하드 차단보다 사전 예상 용량과 경고를 우선한다.

## 10. 전경 추출, 합성, 자동 라벨

### 10.1 MVP 생성 전략

**[제안]** 로컬 생성형 AI보다 결정적 2D 합성을 먼저 채택한다.

~~~mermaid
flowchart LR
    A["원본 대상 이미지 세트"] --> B["로컬 전경 추출 및 사용자 보정"]
    C["사용자·번들 배경 이미지 세트"] --> D["조건 샘플러"]
    E["구조화된 상황·분포·랜덤 시드"] --> D
    B --> F["2D 합성 엔진"]
    D --> F
    F --> G["기하 변환·조명·노이즈·블러·가림"]
    G --> H["최종 가시 마스크로 Bounding Box 계산"]
    H --> I["이미지·라벨·레시피·시드 저장"]
    I --> J["품질 검사 및 사용자 검토"]
~~~

MVP 합성 항목은 위치, 크기, 회전, 기울기, 원근, 밝기, 대비, 색상, 노이즈, 블러, 그림자, 반사, 부분 가림, 주변 객체, 대상 개수, 부정 이미지다.

### 10.2 전경 추출 정책

- 자동 배경 제거는 로컬 모델로만 수행한다.
- 자동 결과에는 신뢰·품질 경고를 표시하고 브러시·지우개·다각형 등 사용자 보정 수단을 제공한다.
- 자동 모델을 확정하기 전 재배포 권리, CPU 속도, Apple Silicon·Windows 결과, 가장자리 품질을 검증한다.
- 자동 모델이 기준을 통과하지 못하면 투명 PNG 입력과 사용자 보정 흐름을 우선 제공하는 축소안을 준비한다.
- 전경 마스크는 내부 합성 자료이며 원본 대상 이미지와 별도 버전으로 저장한다.

### 10.3 Bounding Box 정확성

**[제안]** 합성 전 좌표가 아니라 모든 변환과 가림을 적용한 최종 가시 마스크에서 Box를 계산한다.

- 이미지 경계를 벗어난 영역을 자른 뒤 좌표를 계산한다.
- 완전히 가려진 객체는 라벨에서 제거하거나 생성 실패로 처리한다.
- 최소 가시 면적 미만인 객체는 설정에 따라 경고 또는 제외한다.
- 변환 전 전체 Box, 변환 후 전체 Box, 최종 가시 Box, 가림 비율을 이력으로 저장한다.
- 출력 좌표는 내부 정규형과 픽셀형을 모두 표현할 수 있게 하되 진실의 원천은 하나로 정한다.
- Box가 이미지 밖에 있거나 넓이·높이가 0인 경우 데이터셋 버전을 만들 수 없게 한다.
- 생성 갤러리에서 Box를 직접 수정하면 자동 생성값과 수정값을 함께 보존한다.

### 10.4 데이터 다양성

조건 샘플러는 단순 독립 랜덤이 아니라 목표 분포를 충족하도록 조합을 계획한다. 원본 대상별 사용 비율, 배경, 크기, 위치, 회전, 가림, 조명, 대상 개수, 부정 이미지 비율을 작업 완료 시 실제 분포와 함께 보고한다.

동일한 시드, 원본 자산 버전, 레시피 버전, 엔진 버전으로 같은 합성을 재현할 수 있어야 한다.

## 11. 데이터셋 버전과 분할

### 11.1 버전 원칙

- 데이터셋 버전은 이미지 참조, 라벨, 승인 상태, 분할, 통계, 레시피, 랜덤 시드의 불변 스냅샷이다.
- 수정이 필요하면 기존 버전을 바꾸지 않고 새 버전을 만든다.
- 모델 버전에서 데이터셋 버전으로 역추적할 수 있어야 한다.
- 실제 이미지, 합성 이미지, 부정 이미지의 비율과 출처를 기록한다.
- 사용자 수정 이력과 제외 이유를 통계에 포함한다.

### 11.2 데이터 누수 방지

**[제안]** 같은 원본 대상, 연속 촬영 그룹, 유사 이미지 그룹, 같은 합성 계보가 학습과 검증·테스트에 동시에 퍼지지 않도록 그룹 단위로 분할한다.

합성 이미지만으로 만든 검증 점수가 실제 성능을 대표하지 않으므로, 실제 촬영 이미지 기반의 고정 벤치마크 세트를 별도로 유지한다. 테스트 세트는 모델 선택 과정에서 반복 튜닝에 사용하지 않는다.

### 11.3 데이터셋 생성 게이트

다음 검사를 통과해야 새 버전을 확정한다.

- 모든 자산 파일 존재 및 체크섬 일치
- 모든 라벨 클래스 ID가 프로젝트 클래스와 일치
- Box 좌표와 이미지 크기 유효
- 제외 이미지 미포함
- 분할 간 그룹 누수 없음
- 대상 포함·부정 이미지 최소 수 경고
- 합성·실제·원본별 분포 보고서 생성
- 랜덤 시드와 생성 엔진 버전 기록

## 12. 모델 학습과 평가

### 12.1 학습 흐름

~~~mermaid
flowchart TD
    A["확정 데이터셋 버전"] --> B["장치·디스크·메모리 사전 검사"]
    B --> C["사전 학습 가중치 기반 단일 클래스 전이 학습"]
    C --> D["Epoch·지표·처리량·메모리 이벤트 저장"]
    D --> E{"중단 또는 장애?"}
    E -- "예" --> F["안전 체크포인트와 오류 저장"]
    F --> C
    E -- "아니오" --> G["최종 체크포인트 및 평가"]
    G --> H["범용 모델 형식으로 변환"]
    H --> I["원본 프레임워크와 범용 런타임 결과 비교"]
    I --> J["모델 버전 확정"]
~~~

CPU는 필수 폴백 경로다. NVIDIA GPU와 Apple Silicon 가속은 자동 감지하되, 지원되지 않는 연산이나 런타임 문제 발생 시 사용자가 이해할 수 있는 이유를 표시하고 CPU 경로를 제안한다.

### 12.2 평가 지표

| 범주 | 지표 후보 |
|---|---|
| 탐지 정확도 | Precision, Recall, F1, mAP50, mAP50-95 |
| 운영 품질 | 이미지당 오탐 수, 미탐 수, 대상 있음·없음 판정 정확도 |
| 어려운 조건 | 작은 대상, 가림, 회전, 어두움, 블러별 Recall |
| 성능 | 이미지당 추론 시간, 초당 이미지 수, CPU·GPU 메모리 |
| 안정성 | 실패율, 취소 응답, 체크포인트 복구 성공률 |
| 이식성 | OS·장치별 탐지 개수, 클래스, Box IoU, 신뢰도 차이 |

하나의 보편적 정확도 합격값을 제품 전체에 고정하지 않는다. 대상 종류와 실제 운영 비용이 다르므로 프로젝트별 목표를 두되, VisionForge 자체 릴리스에는 대표 벤치마크 회귀 기준을 둔다.

### 12.3 모델 후보

| 후보 | 장점 | 주요 검증 위험 |
|---|---|---|
| YOLOX Nano/Tiny 계열 | 작은 모델 후보, ONNX 지원 경로, Apache-2.0 코드 | 유지보수 활발성, 사전 가중치 권리, 작은 대상 성능 |
| RTMDet Tiny 계열 | 정확도·속도 균형 후보, OpenMMLab 생태계 | 의존성·패키지 크기, 배포 변환 복잡도, 가중치별 권리 |
| TorchVision 탐지 모델 | PyTorch와의 결합이 단순한 기준선 | CPU 속도, 모델 크기, ONNX 연산 호환성 |

**[미확정]** 모델은 이름이나 공개 벤치마크만으로 결정하지 않는다. 동일한 VisionForge 대표 데이터셋에서 학습 성공률, 실제 이미지 성능, CPU 속도, 메모리, 변환 성공, 교차 플랫폼 결과, 코드와 가중치의 재배포 권리를 함께 평가한다.

## 13. 추론과 피드백 루프

### 13.1 추론 파이프라인

1. 모델 패키지와 입력 이미지 세트를 선택한다.
2. 패키지 무결성, 호환성, 저장 공간, 실행 장치를 검사한다.
3. 이미지별로 동일한 디코딩·방향·색상·리사이즈·정규화를 적용한다.
4. 가능한 가속 실행 공급자를 선택하고 CPU 폴백을 유지한다.
5. 고정된 NMS, 임계값, Box 변환, 정렬 규칙을 적용한다.
6. 이미지별 탐지, 처리 시간, 장치, 오류를 저장한다.
7. Box가 표시된 결과 이미지는 원본과 별도 자산으로 렌더링한다.
8. 목록·갤러리에서 다중 검토, 상태 변경, 재추론, 내보내기를 제공한다.

### 13.2 이미지 결과 상태

대상 있음, 대상 없음, 여러 개 있음, 낮은 신뢰도, 검토 필요, 오탐 의심, 미탐 의심, 처리 실패, 사용자 승인, 사용자 수정, 재학습 후보, 재학습 포함 상태를 조합 가능한 필드로 저장한다. 하나의 문자열 상태에 모든 의미를 압축하지 않는다.

이 상태 분류의 근거는 해당 이미지의 Detection 목록, 임계값, 오류, 사용자 검토 결과다. MVP에서는 이를 위해 별도의 이미지 분류 모델을 학습하지 않는다.

### 13.3 재학습

- 오탐 이미지는 부정 이미지 또는 잘못된 Box 수정 데이터로 추가한다.
- 미탐 이미지는 사용자가 Box를 추가한 뒤 양성 데이터로 추가한다.
- 낮은 신뢰도 이미지는 검토 후 포함 여부를 결정한다.
- 기존 데이터셋과 모델을 수정하지 않고 새 버전을 만든다.
- 이전 모델과 새 모델을 같은 고정 평가·추론 세트에서 비교한다.
- 사용자가 채택한 모델만 프로젝트의 기본 추론 모델로 지정한다.

## 14. 모델 이식성과 .vfmodel

### 14.1 이식성 기준선

**[제안]** 학습 체크포인트와 배포 모델을 분리한다.

- 학습 재개용 체크포인트는 프로젝트 내부에 보관한다.
- 공유용 패키지는 범용 추론 모델을 중심으로 구성한다.
- ONNX를 1차 범용 형식 후보로 검증한다.
- ONNX Runtime CPU 실행을 호환성 기준선으로 둔다.
- Windows NVIDIA CUDA와 macOS Core ML 계열 가속은 선택 실행 공급자로 검증한다.
- 가속용으로 변환·컴파일된 캐시는 각 컴퓨터에서 생성하며 공유 패키지의 진실의 원천으로 삼지 않는다.

ONNX Runtime은 동일 API에서 CPU와 여러 하드웨어 실행 공급자를 선택할 수 있지만, 공급자별 연산 지원과 결과 차이는 반드시 모델별 호환성 시험을 거쳐야 한다.

### 14.2 제안 패키지 구조

~~~text
VisionForgeModel.vfmodel
├── manifest.json
├── model/
│   └── model.onnx
├── labels.json
├── pipeline/
│   ├── preprocessing.json
│   └── postprocessing.json
├── provenance/
│   ├── training-info.json
│   ├── dataset-summary.json
│   └── metrics.json
├── compatibility.json
├── licenses/
│   ├── model-license.json
│   └── third-party-notices.txt
├── preview.png
└── integrity/
    ├── checksums.json
    └── signature.json
~~~

패키지는 ZIP 계열 컨테이너처럼 하나의 파일로 전달하되 확장자, 매직 헤더, 스키마 버전, 압축 해제 한도, 경로 탈출 방지 규칙을 별도로 정의한다.

### 14.3 manifest 필수 항목

| 범주 | 정보 |
|---|---|
| 식별 | 패키지·모델 ID, 이름, 설명, 버전, 생성 시각 |
| 앱 호환 | 생성 VisionForge 버전, 최소 호환 버전, 패키지 스키마 |
| 모델 | 형식, 해시, opset, 입력·출력 Tensor 정의, 필요한 연산자 |
| 클래스 | ID, 이름, 순서, 설명 |
| 입력 | 너비·높이, 색상 순서, 알파, EXIF 방향, 가변 크기 여부 |
| 전처리 | 리사이즈, 종횡비, 보간, Letterbox, Padding, 정규화 |
| 후처리 | 신뢰도, IoU, NMS 방식, 최대 탐지 수, 좌표·정렬 규칙 |
| 호환성 | OS, CPU 아키텍처, 실행 공급자, 최소 런타임 |
| 출처 | 데이터셋 버전, 초기 가중치, 학습 설정 요약, 지표 |
| 권리 | 코드·런타임·사전 가중치·사용자 모델 라이선스 |
| 무결성 | 파일별 SHA-256, 전체 패키지 식별, 선택적 서명 |

### 14.4 가져오기 검증 순서

1. 확장자와 매직 헤더를 확인한다.
2. 압축 파일 크기, 항목 수, 경로, 압축률 제한을 검사한다.
3. manifest 스키마와 필수 파일을 검사한다.
4. 파일별 체크섬을 검사한다.
5. 앱·OS·CPU·실행 공급자·opset·연산자 호환성을 검사한다.
6. 모델을 격리된 검증 세션에서 열고 입력·출력 계약을 확인한다.
7. 포함된 소형 검증 벡터가 있다면 전처리부터 후처리까지 실행한다.
8. 동일 ID·버전 충돌 정책과 저장 공간을 확인한다.
9. 가속 공급자가 불가능하면 CPU 사용 가능 여부를 제시한다.
10. 사용자가 클래스, 출처, 라이선스, 호환성 요약을 확인한 뒤 가져온다.

내보내기 완료 후에도 생성된 패키지를 새 파일처럼 다시 열어 위 검사를 수행한다.

### 14.5 기능적 동일성

필수 기준은 다음과 같다.

- 동일한 클래스와 클래스 순서
- 동일한 입력 해석과 전처리
- 동일한 신뢰도·IoU 임계값
- 동일한 NMS와 결과 정렬
- 동일한 대상과 탐지 개수
- 실질적으로 같은 위치의 Box
- 동일한 대상 있음·없음 판정

**[제안]** 전처리와 후처리를 플랫폼 네이티브 이미지 API에 맡기지 않고 VisionForge의 고정 구현으로 통일한다. 가속 모드에서도 NMS와 최종 정렬은 가능한 한 공통 CPU 후처리를 사용한다.

교차 플랫폼 검증은 동일 이미지 코퍼스에 대해 탐지 개수·클래스 일치, Box IoU, 신뢰도 차이를 비교한다. 허용 수치는 PoC 결과로 확정한다. 임계값 근처 결과는 하드웨어 오차로 상태가 바뀔 수 있으므로 경계 신뢰도 경고를 별도로 검토한다.

### 16.2 오프라인 업데이트

업데이트는 파일 선택, USB, 외장 저장 장치, 사내 파일 서버 경로에서 수행한다.

1. 업데이트 manifest, 대상 앱 버전, OS·아키텍처를 검사한다.
2. 패키지 체크섬과 서명 정책을 검사한다.
3. 필요한 디스크 공간과 프로젝트 스키마 영향을 미리 계산한다.
4. 앱 설정과 프로젝트 DB를 백업한다.
5. 앱 파일은 임시 위치에 설치 후 원자적 전환 또는 설치 관리자 트랜잭션을 사용한다.
6. 프로젝트 마이그레이션은 복사본에서 사전 검사한다.
7. 실행 검증 실패 시 이전 앱 버전과 프로젝트 백업으로 되돌린다.
8. 구버전 모델과 프로젝트의 읽기·변환 정책을 호환성 표로 관리한다.

온라인 자동 업데이트는 후속 선택 기능이며, 수동 오프라인 업데이트 경로가 항상 존재한다.

프로젝트 암호화, 모델 암호화, 비밀번호, 조직 인증서, 프로젝트별 접근 제어, 미리보기 제한은 위협 모델과 고객 환경을 확인한 뒤 범위를 확정한다. 운영체제 앱 서명과 배포 인증은 기업 보안 기능과 별개로 릴리스 배포 요건에서 검토한다.

## 18. 기술 후보와 선택 전략

### 18.1 데스크톱 UI

| 후보 | 적합한 이유 | 위험·검증 항목 |
|---|---|---|
| PySide6 + Qt Quick | Python AI 계층과 언어·도구 통합이 쉬움, Windows·macOS 배포 도구 제공 | LGPLv3·상용 라이선스 검토, 번들 크기, QML 성능, 패키징 재현성 |
| Tauri + 웹 UI + 로컬 AI sidecar | UI와 AI 프로세스 경계가 명확하고 MIT·Apache 계열 코드, 운영체제별 설치 번들 | Rust·웹·Python의 다중 스택, 아키텍처별 sidecar, WebView2 오프라인 포함 정책 |

**[제안]** 두 후보 모두 같은 로컬 작업·저장·AI 계약을 사용하는 얇은 기술 PoC로 비교한다. 소규모 팀의 MVP 속도가 최우선이면 PySide6를 1차 후보로, UI 확장성과 명확한 프로세스 보안 경계가 우선이면 Tauri를 1차 후보로 둔다. 최종 결정은 실제 설치 패키지 크기, 1,000개 썸네일 스크롤, 작업 이벤트 처리, 비정상 종료 복구, 라이선스 검토 결과로 내린다.

### 18.2 AI 및 저장 기준선

| 영역 | 1차 후보 | 대안·주의 |
|---|---|---|
| 학습 | PyTorch 기반 로컬 학습 | Apple MPS 연산 지원과 체크포인트 호환성 검증 |
| 공유 추론 | ONNX + ONNX Runtime CPU | 공급자별 연산 차이와 opset 고정 |
| Windows 가속 | ONNX Runtime CUDA 후보 | CUDA·cuDNN·드라이버 호환 매트릭스 |
| macOS 가속 | ONNX Runtime Core ML 후보 | 연산 지원, preview 상태, CPU 대비 결과 검증 |
| 이미지 처리 | 고정 버전의 교차 플랫폼 디코더·OpenCV 계열 후보 | 색상, EXIF, 보간의 플랫폼 차이 차단 |
| 메타데이터 | SQLite | WAL, 백업, 마이그레이션, 파일·DB 일관성 |

이 표는 기술 확정이 아니라 PoC 시작점이다. 각 코드 라이선스와 사전 학습 가중치, 포함 데이터, 배포 바이너리의 권리를 별도로 확인한다.

### 18.3 로컬 LLM 검토 결론

**[제안] MVP 제외.** 배경, 조명, 각도, 거리, 크기, 위치, 회전, 가림, 개수, 노이즈, 블러, 시간대, 날씨를 구조화된 옵션과 프리셋으로 제공하면 핵심 흐름을 완성할 수 있다.

후속 버전에서 자연어 입력 수요가 확인되면 규칙·키워드 파서를 먼저 검증하고, 그 한계가 명확할 때만 선택형 소형 로컬 LLM 패키지를 검토한다.

### 18.4 로컬 생성형 이미지 모델 검토 결론

**[제안] MVP 제외.** 생성형 모델은 대상 형태·로고·세부 특징을 바꿀 수 있고 자동 Box의 신뢰도를 낮추며 설치 용량과 하드웨어 요구를 크게 높인다.

MVP는 원형 보존과 라벨 재현성이 높은 2D 합성으로 제품 가치를 검증한다. 합성-실사 격차가 핵심 한계로 확인된 뒤에만 선택형 생성 모델 또는 인페인팅을 검토한다.

## 19. 구현 전 PoC 계획

기술 스택 확정보다 제품 실패 위험이 큰 순서로 검증한다. 각 PoC는 날짜가 아니라 통과 조건으로 종료한다.

### PoC 1. 합성 이미지와 Box 정확성

목표는 여러 원본 대상, 배경, 회전, 원근, 이미지 경계, 가림 조건에서 최종 가시 Box가 정확한지 검증하는 것이다.

통과 조건:

- 자동 Box가 최종 가시 대상과 일치
- 같은 시드와 엔진 버전에서 결과 재현
- 이미지 1장과 N장이 같은 경로로 처리
- 중간 실패 후 완료 항목 재사용

### PoC 2. 합성 데이터의 실제 탐지 성능

목표는 합성 데이터만 또는 소량 실제 데이터 혼합으로 실제 촬영 이미지에서 유효한 탐지가 가능한지 확인하는 것이다.

통과 조건:

- 대표 대상 최소 3종에서 고정 실제 평가 세트 구축
- 합성 조건별 성능과 실패 조건을 설명 가능
- 부정 이미지 추가가 오탐 감소에 미치는 영향 측정
- 작은 대상·가림·회전의 최소 실용 수준 확인

이 PoC가 실패하면 생성 모델을 바로 추가하지 않는다. 전경 품질, 배경 분포, 실제 이미지 비율, 증강 강도, 모델 후보를 먼저 조정한다.

### PoC 3. 학습 모델 이식성

목표는 한 OS에서 학습한 모델을 ONNX 후보로 변환해 다른 OS와 장치에서 기능적으로 동일하게 실행하는 것이다.

통과 조건:

- Windows와 macOS에서 같은 .vfmodel 열기
- CPU 호환 모드에서 탐지 개수·클래스·Box가 허용 범위 내 일치
- NVIDIA·Apple 가속 실패 시 CPU 폴백
- 지원되지 않는 연산자와 런타임 버전을 가져오기 전에 탐지

### PoC 4. 장시간 작업 복구

목표는 합성, 학습, 추론 중 강제 종료·재부팅·디스크 부족을 모사해 복구 구조를 검증하는 것이다.

통과 조건:

- DB와 파일 불일치 자동 탐지
- 완료 항목 재사용, 실행 중 항목만 재처리
- 취소 후 프로젝트 재사용 가능
- 손상 없는 백업·복원

### PoC 5. 오프라인 설치와 패키지 크기

목표는 깨끗한 Windows 및 macOS 컴퓨터에서 네트워크 없이 설치·학습·추론하는 것이다.

통과 조건:

- Python·개발 도구 없는 컴퓨터에서 설치
- 기본 모델과 CPU 런타임 포함
- 네트워크 차단 상태의 전체 핵심 흐름
- 운영체제별 업데이트와 롤백

### PoC 6. UI 프레임워크 비교

목표는 PySide6와 Tauri 중 제품 구조에 맞는 셸을 선택하는 것이다.

통과 조건:

- 1,000개 이상 썸네일의 가상 스크롤·필터
- Box 오버레이와 편집 반응성
- 작업 진행 이벤트와 앱 재시작 복구
- 설치 패키지 재현성
- 상용 배포 라이선스 검토 완료

| 7. 배포 안정화 | 전체 설치, 가속 패키지, 업데이트, 롤백 | 기능 동결 | 완전 오프라인 수용 시험 통과 |

실제 개발 일정은 PoC 결과, 지원 하드웨어, 테스트 장비, 인력 구성이 확정된 뒤 산정한다.

### 21.2 MVP 종료 기준

MVP는 아래 시나리오를 모두 만족할 때 완료로 본다.

1. 네트워크가 차단된 Windows와 macOS에서 설치 후 실행된다.
2. 대상 이미지 여러 장을 등록하고 품질 경고를 검토할 수 있다.
3. 2D 합성 이미지와 부정 이미지를 여러 장 만들고 Box를 수정할 수 있다.
4. 불변 데이터셋 버전으로 로컬 모델을 학습한다.
5. 추론 이미지 여러 장 중 일부가 손상돼도 나머지 결과가 생성된다.
6. 이미지별 대상 존재, 개수, 위치, 신뢰도, 오류가 저장된다.
7. 오탐·미탐을 새 데이터셋 버전에 반영하고 모델을 비교할 수 있다.
8. 모델을 .vfmodel로 내보내고 새 컴퓨터에서 무결성·호환성을 검사한다.
9. 다른 OS에서 재학습 없이 같은 이미지 세트를 기능적으로 동일하게 추론한다.
10. 장시간 작업 중 앱을 종료해도 체크포인트에서 복구된다.
11. CSV, JSON, 결과 이미지를 일괄 내보낼 수 있다.
12. 실행 중 이미지·모델·로그가 외부 서버로 전송되지 않는다.
